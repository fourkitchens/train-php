Making a Module and Theme

Module's .info File

Every module needs a .info file to alert Drupal that it exists. The .info file requires you have:

  • Name of the module
  • Description of the module
  • Which version of Drupal the module is for

Module's .info File

What it looks like right now


name = My Very Own Module
description = A module for the 2012 BAD camp PHP training.
core = 7.x

Module's .info File

You can also include the following:

  • Package (e.g. views)
  • Dependencies
  • Stylesheets
  • Javascript files
  • PHP Version
  • Configuration page URL

The Hook System

Module's .module File (reprise)

NOW we can start writing our code, and to do that we are going to use some new hooks. Namely hook_menu() and hook_permission().

Both of these hooks will be found in 99% of each module that is coded.

hook_menu()

This hook will allow you to create your own pages on your Drupal site. You pass back an array, where the keys are the URLs that you are assigning.


function YOUR_MODULE_menu() {
  $items = array();

  $items['your/url/in/full'] = array();

  return $items;
}

hook_menu()

For hook_menu to work, you must return within your array. Here are those variables:


function YOUR_MODULE_menu() {
  $items['your/full/url'] = array(
    'title' => t(''),
    'page callback' => '',
    'type' => MENU_NORMAL_ITEM,
    'access callback' => '',
    'access arguments' => array(''),
  );

  return $items;
}

hook_menu()


function YOUR_MODULE_menu() {
  $items['your/full/url'] = array(
    'title' => t('This is my site.'),
    'page callback' => 'YOUR_MODULE_your_function_name',
    'type' => MENU_NORMAL_ITEM,
    'access callback' => 'user_access',
    'access arguments' => array('??????'),
  );

  return $items;
}

What is this access thing?

This is where hook_permission comes into play...

hook_permission()

hook_permission is how you tell Drupal the permissions that your module needs to use. It allows you to assign certain role groups permission to use or not use features within your module.

After you set your own permissions, you can use user_access to check and see if the current user has permissions to access that section of your code.

hook_permission()


function YOUR_MODULE_permission() {
  return array(
    '' => array(
      'title' => t(''),
      'description' => t(''),
    ),
  );
}

hook_permission()


function YOUR_MODULE_permission() {
  return array(
    'permission name' => array(
      'title' => t('What you want to call it'),
      'description' => t('A description to be shown on Drupal\'s permission page'),
    ),
  );
}

So, within hook_menu() access arguments, we would put 'permission name'.

Both together


function YOUR_MODULE_menu() {
  $items['your/full/url'] = array(
    'title' => t('This is my site.'),
    'page callback' => 'YOUR_MODULE_your_function_name',
    'type' => MENU_NORMAL_ITEM,
    'access callback' => 'user_access',
    'access arguments' => array('permission name'),
  );

  return $items;
}
        

function YOUR_MODULE_permission() {
  return array(
    'permission name' => array(
      'title' => t('What you want to call it'),
      'description' => t('A description to be shown on Drupal\'s permission page'),
    ),
  );
}

Ignore that one... A theme is just a module, with the specific goal of assigning the front end structure to the site. It is still made up by the same basic parts.

A .info file, to alert Drupal of our theme

But instead of a .module file, we use a template.php file to do any PHP within.

.module
VS.
template.php

A .module is for modules, and can contain any hooks

The template.php is just for theme preprocessors (mainly) and usually doesn't contain non-theme hooks
